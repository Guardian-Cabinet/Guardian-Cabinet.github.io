## 📈 Google Analytics (GA4) 데이터의 의미 있는 활용 방안

기존의 `visits.json` 자체 트래킹을 넘어 GA를 결합하면 구직 및 프리랜서 활동에 있어 엄청난 무기가 될 수 있습니다.

**1. 이탈률과 탭 전환 분석 (UX 최적화)**
- **의미**: 방문자(인사담당자, 고객)가 'PROFILE' 탭에서 'PORTFOLIO' 탭으로 넘어가는 비율, 혹은 어떤 탭(예: SKILLS)에서 가장 많이 이탈하는지 파악 가능.
- **활용**: 이탈이 많은 탭의 디자인을 수정하거나, 포트폴리오 첫 화면에서 가장 매력적인 역량을 위로 올리는 A/B 테스트 근거 자료가 됨.

**2. 유입 채널 분석 (Targeting)**
- **의미**: GA의 획득 리포트를 통해 방문자가 '이력서 링크(Direct)', 'LinkedIn', '카카오톡', '깃허브' 중 어디로 들어왔는지 파악 가능.
- **활용**: 링크드인을 통한 유입자 체류시간이 높다면 링크드인 프로필 업데이트에 집중하고, 직접 링크 복사로 들어오는 트래픽이 많다면 오프라인 숏링크(QR) 효과가 높다는 뜻이므로 그부분 지원.

**3. 페이지 체류 시간 (콘텐츠 매력도)**
- **의미**: 포트폴리오 동영상 재생 버튼을 클릭했는지, 페이지에 얼마나 머물렀는지 추적.
- **활용**: 체류 시간이 10초 미만이라면 텍스트가 너무 많다는 뜻. 체류시간이 길다면 포트폴리오 영상이나 이미지에 관심이 크다는 뜻으로 영상 콘텐츠를 강화하는 전략 수립.

**4. 기기 및 해상도 데이터 (Responsive Polish)**
- **의미**: 접속자의 80%가 모바일인지, 혹은 PC인지 파악.
- **활용**: 현재 UI가 폴더 탭 형식인데 모바일 접속자가 많다면 모바일 탭 디자인이나 스크롤 편의성에 개발 리소스를 더 투입해야 함.

## 🛠️ GitHub Pages 미반영 원인 분석 보고서

현재 `http://localhost:8081/admin`에서 수정한 내용이 `https://guardian-cabinet.github.io/`에 반영되지 않는 원인을 카테고리별로 분석한 결과입니다.

| 카테고리 | 상세 원인 | 분석 내용 | 해결 방안 |
| :--- | :--- | :--- | :--- |
| **1. 배포 프로세스 중단** | **Submodule 에러** | `portfolio_server` 폴더 내부에 `.git` 설정이 남아있어 GitHub이 이를 '서브모듈'로 인식함. 배포 액션 실행 시 해당 경로를 찾지 못해 전체 빌드가 중단됨. | `portfolio_server` 내 `.git` 제거 후 일반 폴더로 강제 인식 처리 |
| **2. 동기화 누락** | **로컬 푸시(Push) 미작동** | 최근 7일간 GitHub에 올라온 커밋이 없음. 로컬의 `main` 브랜치 내용이 GitHub(`upstream`)으로 전달되지 않은 상태임. | 로컬 변경사항(index.html, data.json 등)을 수동으로 `push` 수행 |
| **3. 환경 설정 오류** | **Jekyll 처리 간섭** | GitHub은 기본적으로 Jekyll 빌드를 시도함. 현재 프로젝트는 대용량 파일(`.venv`)이 포함되어 있어 Jekyll 빌드 중 오류가 발생할 확률이 높음. | `.nojekyll` 파일을 생성하여 GitHub의 자동 빌드 프로세스 우회 |
| **4. 리모트 명칭 불일치** | **upstream vs origin** | 서버의 자동 배포 코드는 `git push`를 시도하지만, 현재 등록된 리모트 이름이 `upstream`이라서 기본 명령어가 작동하지 않을 수 있음. | `main.py`의 푸시 로직을 명확하게 수정하거나 리모트 별칭 정리 |
| **5. 캐시 문제** | **브라우저/CDN 캐시** | GitHub 배포가 성공하더라도 브라우저가 이전 버전(v3.0)을 기억하고 있을 수 있음. | 강력한 새로고침(Ctrl + F5) 및 GitHub 배포 완료 확인 필요 |
