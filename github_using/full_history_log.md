# Full Project History Log: Digital Archive Portfolio (2026-04-01)

본 로그는 프로젝트의 시작부터 현재의 GitHub 배포 준비 단계까지 모든 주요 지시 사항과 구현 내용을 정리한 기록입니다.

---

## 🕒 [Phase 1] 기획 및 디자인 컨셉 정의
*   **요청:** GitHub 홈페이지 제작을 위한 네이밍 및 디자인 테마 제안.
*   **결정:** '파일 캐비닛/디지털 아카이브' 컨셉 채택. 폴더를 열고 닫는 물리적 인터페이스 구현.
*   **디자인 고도화:**
    *   `162131.png` 레퍼런스: 상단 블랙 헤더 라인, 굵은 테두리(Bold Border), 지그재그식 탭 배치 적용.
    *   `161142.png` 레퍼런스: CAREER 섹션에 수직형 타임라인(인포그래픽) 스타일 적용.

## 🚀 [Phase 2] 기술적 전환 (Render → GitHub Pages)
*   **배경:** 기존 Render 서버의 느린 초기 로딩 속도 문제를 해결하기 위해 고속 정적 호스팅인 GitHub Pages로 이전 결정.
*   **전환 전략:** 서버(Python) 없이 깃허브에서만 동작하도록 모든 데이터를 JSON 파일로 변환 및 JS 기반 엔진 구축.
*   **기능 유지:** 서버 없이도 **방문자 추적(Visits)**과 **관리 도구(Admin)**를 사용할 수 있도록 클라이언트 사이드 로직 및 Firebase/Static JSON 구조 설계.

## 🛡️ [Phase 3] 보안 시스템 고도화
*   **세분화된 노출 제어:** `admin.html`의 체크박스를 통해 개별 프로젝트나 섹션의 노출 여부를 실시간으로 제어(Visibility Matrix).
*   **접속 기간 제한:** 관리자가 설정한 기간(시작일~종료일) 동안만 사이트가 열리도록 타임-밤(Time-Bomb) 로직 구현.
*   **2단계 인증 (ID/PIN):** 단순히 비밀번호만 입력하는 것이 아니라, 허용된 아이디와 비번 쌍을 아는 사람만 들어올 수 있도록 `security.js` 강화.

## ⚙️ [Phase 4] 관리 및 데이터 통합
*   **로컬 관리 도구 (`admin.html`)**: 사용자가 본인 컴퓨터에서 직접 설정을 관리하고 `config.json`을 내보낼 수 있는 대시보드 구축.
*   **데이터 완전 이식:** `portfolio_server`의 원본 데이터(14KB)를 분석하여 `content.json`으로 100% 이식 완료.
*   **복구 전략:** 로컬 관리 파일 자체가 '마스터 키' 역할을 하여 비번 분실 시에도 언제든 재설정 가능하도록 설계.
